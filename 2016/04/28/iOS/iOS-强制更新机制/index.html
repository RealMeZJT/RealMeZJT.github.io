iOS 强制更新机制

发表于 2016-04-28 | 分类于 iOS How-To

为你的app预留一个强制更新的功能总是没错的。
就我而言，就遇到过这么些情况。
有比较任性的：“嘿，我们这个版本有重大的更新，为了让用户体验到我们的最新功能，请强制他们更新”；

有后台更新请求接口的：“这个操作的请求接口现在需要app多传这么些参数给后台，以配合另一个新功能。等新版app上线后，我们就将新的后台代码放到线上，而app端需要强制用户更新到最新版”。

.
.
.

强制更新很可能会惹恼你的用户,请慎用（当然，往往不由得我们决定，而是项目负责人说了算）。

接着我们来探讨，有哪些强制更新的机制：

利用消息推送

需要后台的配合。app端和后台约定好，消息推送传过来的json数据，什么字段用于判断强制更新，该字段什么值时表示需要进行强制更新。
当需要强制更新的时候，后台向所有设备发送一次推送消息。当app端接收到该推送消息的时，弹窗提示用户进行更新。值得注意的是，后台只发送一次推送，为了保证下次打开app时，仍能检查到需要更新最新版，你需要将用到缓存。然后在 AppDelegate的applicationDidBecomeActive方法内获取缓存数据检测是否强制更新。

比如：你可以收到强制更新的推送时，就将当前版本号缓存起来 NSUserDefaults.standardUserDefaults().setObject(currVersion, forKey:"theVersionNeedUpdate")
然后在AppDelegate的applicationDidBecomeActive方法内判断”theVersionNeedUpdate”与当前版本是否相等，是的话，表示还没更新。如果不相等就表示安装了新版本。

func applicationDidBecomeActive(application: UIApplication) {
        let theVersionNeedUpdate = NSUserDefaults.standardUserDefaults().stringForKey("theVersionNeedUpdate")
        if  currVersion == theVersionNeedUpdate {
            showForceUpdateAlert()
        }
}

使用此机制，后台可以很及时的控制app强制更新。当然，致命的弊端就是：如果用户禁止消息推送，就作废了🤕。
因此该机制只能作为辅助手段。

利用itunes版本号

一般使用的是GNU 风格的版本号：主版本号.子版本号.[修正版本号]

我们为app设置的版本号可以通过这条URL获取到：https://itunes.apple.com/lookup?id=<Apple ID>
当你在itunesconnect创建应用时，就有一个固定的Apple ID与该应用对应。在itunesconnect上可查看：

该URL返回的json数据，内有version字段，对应的值就是该app在App Store上的最新版本号。
所以，有这么一个主意，通过对比主版本号来标识是否需要进行强制更新。app通过对比返回数据的主版本号和当前app的主版本号，当前者比后者大时，进行强制更新。
比如，App Store上现有版本是2.0.0，现在发布一个新版本，需要旧版本强制提示更新，那么新版本号该改为3.0.0。当新版本审核通过后，从iTunes获取到的版本号将变为3.0.0。旧版本app的主版本号低于App Store上的主版本号，就会判断到需要强制提示更新。

使用此机制的好处是，不依赖后台。
坏处是网络状况不佳的情况下，检测更新不及时。

配合后台接口

规定每个网络请求都传递app的内部版本号，后台根据app版本号判断是否需要强制更新，如果需强制更新，则向此次请求返回约定好的错误信息，app端接收到该错误信息，提示用户更新。
好处是 检测更新及时。
坏处是 需要后台配合，对于旧项目，后台还需要为旧接口补充判断强制更新的逻辑。
